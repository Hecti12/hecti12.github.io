Hardware & Driver Troubleshooting | NVIDIA RTX 5080

RTX 5080 Black-Screen & Driver Stability Fix

Diagnosed and resolved repeated black screens and lost display signal on a dual extreme-refresh setup after an NVIDIA 596.49 driver update on a Blackwell-architecture RTX 5080, then documented the full process and shared it publicly to help others hitting the same bug.

check_circle Result: Idle VRAM clock dropped from a pinned 15001 MHz back to a normal 7001 MHz, core temperatures fell, and the system now runs 100% stable.
expand_moreRead the full step-by-step troubleshooting writeup

Problem

  • PC began randomly black-screening and dropping the display signal under load immediately after updating to NVIDIA driver 596.49.
  • Windows "View reliability history" showed a long string of hardware crashes logged as LiveKernelEvent 1b8 and 193 codes.
  • GPU VRAM clocks were permanently pinned at maximum (15001 MHz) even at 0% desktop load, severely hurting stability.
  • Confirmed a documented bug on the 596.49 branch for Blackwell cards: configurations demanding over a 1620 MHz pixel clock trip a high-bandwidth threshold that disables GPU scaling and forces strict Display Stream Compression (DSC) routing, a branch the community flagged for idle timing timeouts and black-screen loops on dual extreme high-refresh setups.

What I did (in order)

  1. First DDU attempt. Ran Display Driver Uninstaller to wipe the graphics path, but initially only selected "NVIDIA" under the GPU dropdown, which left broken driver remnants behind in the system cache.
  2. Recovered the black screen, then cleaned properly. With only the generic Windows fallback driver, the displays went black because it could not handshake the panels' bandwidth demands. I forced a basic 60 Hz signal on a single monitor over HDMI to get the desktop back, then re-ran DDU and selected GPUs → ALL to fully scrub the driver stack.
  3. Updated monitor firmware. With the driver path cleared, I updated the internal firmware on the Alienware monitors using a USB upstream (printer) cable connected directly to the PC, as the firmware notes specify, to resolve connection and handshake issues.
  4. Swapped GPU power cabling. Replaced the bulky 3× 8-pin split adapter on the RTX 5080 with a native 12-pin-to-two-8-pin PCIe cable from an identical, matching PSU model. PSA: never mix cables across different power supplies, pinouts are not universal and a mismatch can fry hardware.
  5. Clean driver reinstall. With the power lines secure and matched, I performed a completely fresh, clean installation of the newest graphics driver.
  6. Matched the refresh rates. Because the new monitor firmware clashed with the driver's high-bandwidth DSC timing at absolute max bandwidth, I dropped the main panel from 480 Hz to 360 Hz in Windows Advanced Display settings so both panels run a steady, matched 360 Hz.

Outcome

Locking both panels to a matching 360 Hz baseline relieved the stress on the display-engine pipeline. The idle VRAM clock instantly cut in half (down to a resting 7001 MHz), core temperatures dropped, and everything is running 100% stable.

Networking Lab | DNS Troubleshooting

DNS Traffic Analysis Lab

Documented a repeatable DNS troubleshooting workflow for help desk and systems support, isolating name resolution failures from broader network problems, then proving the fix with live packet capture by following a single query end to end, from client to resolver and back.

  • Worked the resolution path top-down: confirmed the client's DNS server and connection-specific suffix with ipconfig /all, tested reachability with ping, and queried records directly with nslookup.
  • Cleared stale cache entries using ipconfig /flushdns and re-tested, separating local caching problems from authoritative resolution failures.
  • Captured live DNS queries and responses in Wireshark, reading source, destination, queried domain, record type, and returned IP to pinpoint exactly where resolution broke down.
  • Mapped common symptoms such as NXDOMAIN, query timeouts, and wrong IP answers to likely root causes and next steps, then wrote support-style documentation so a Tier 1 technician can repeat the workflow and escalate with evidence.
